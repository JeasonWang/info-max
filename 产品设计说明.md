# 信息达人前端设计说明

## 1. 产品定位

`Info Max` 面向普通用户，目标是把后端 `Info_aggregation` 的多渠道热点信息，以更低理解成本的方式展示出来。设计重点不是“功能堆叠”，而是：

- 首屏快速理解当前全局热点
- 快速按分类 / 渠道 / 关键词筛选
- 列表浏览与详情阅读切换顺畅
- 保留后续扩展图片、视频、Markdown、HTML 等内容格式的能力

## 2. 页面结构

### 首页看板

- 顶部 Hero：明确产品定位与系统连接状态
- 实时统计：展示总量与分类分布
- 筛选器：分类、渠道、关键词
- 内容列表：主阅读区，支持进入详情
- 时间线：强调“最新事件流”
- 渠道运维：支持直接触发后端爬取

### 详情页

- 顶部信息摘要：标题、分类、渠道、抓取状态
- 元数据面板：时间、主体、地点、指标
- 正文内容：根据内容特征做不同展示策略
- 来源跳转：回到原始页面

## 3. 交互设计

- 列表页保留明显筛选入口，减少层级跳转
- 详情页单独路由化，便于 Web 分享和 H5 打开
- 手动触发爬取放在渠道面板中，避免污染用户主阅读流
- 时间线采用轻量侧栏，不与主列表争抢视觉焦点

## 4. UI设计方向

- 风格：浅色、信息型、轻玻璃拟态
- 色彩：暖橙色作为强调色，配合浅米白背景，避免传统后台系统的生硬感
- 布局：桌面端双栏，移动端单栏
- 组件：统一卡片、胶囊标签、圆角按钮，降低视觉噪音

## 5. 多端扩展建议

当前实现优先保证 Web / H5 可运行，后续迁移到 `uni-app` 时建议保持：

- `types.ts` 继续作为统一数据契约
- `services/api.ts` 抽离为跨端请求层
- 页面结构保持 “首页看板 + 详情页” 两层模型
- 内容渲染组件按端能力分别实现

## 6. 为什么这样做

需求文档中提到要兼顾多渠道、多内容类型和多端显示。如果一开始直接把小程序、H5、Web 三端同时做完，工程复杂度会显著升高，而且后端当前接口仍偏内容查询型。因此本次实现先把：

- 数据模型
- 页面信息结构
- API 对接边界
- 响应式设计

先稳定下来。这部分才是后续扩展到 `uni-app` 和微信小程序最有价值的基础设施。
